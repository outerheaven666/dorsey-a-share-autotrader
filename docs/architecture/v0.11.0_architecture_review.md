# v0.11.0 Architecture Review

## 当前定位

v0.11.0 是一个 research-only、local CSV、mock provider、paper/backtest simulation、dry-run notification、safety-gated、release-checkable 的研究型系统骨架。

它不是实盘系统，不提供投资建议，不保证收益，不支持真实交易，不连接真实券商，不连接真实网络数据源，也不保存真实 broker credentials。

看起来能跑，不代表可以交易。当前最重要的价值不是“已经有很多模块”，而是安全边界、可复验流程、审计记录和数据接入前的防错结构。

## 当前系统总览

- `scoring`: 基于质量、护城河、估值、风险的规则化打分。
- `portfolio construction`: 基于 composite score 的目标组合生成。
- `paper broker`: 本地模拟成交和交易记录。
- `backtest`: 季度调仓、本地历史样例数据、交易成本和 A 股交易限制模拟。
- `data quality`: 缺失值、异常值、stale data、防未来函数检查。
- `schema validation`: 本地 CSV schema、字段类型、重复键和额外字段检查。
- `point-in-time data`: 通过 `disclosure_date <= as_of_date` 控制财务数据可见性。
- `factor audit`: 因子级分解，解释每个股票得分来源和红旗拦截原因。
- `adapter contract`: mock provider contract、字段映射和 schema 校验。
- `schema versioning`: provider contract YAML、baseline/candidate diff。
- `schema migration`: 字段迁移、deprecation 生命周期、兼容窗口和迁移说明。
- `pre-live safety gate`: 默认阻断 live trading、real broker、real order、real network data。
- `system health`: 本地系统健康检查、敏感内容扫描、artifact manifest。
- `release checklist`: v0.11.0 发布候选清单和 release notes draft。

## MVP 1-11 回顾

### MVP 1: 基础骨架、评分、组合、paper broker

做了什么：

- 建立 Python `src` layout。
- 定义股票、财务、市场、因子、组合等数据模型。
- 实现 red-flag、quality、moat proxy、valuation、composite score。
- 实现目标组合构建和本地 paper broker。
- 提供 `run-score`、`build-portfolio`、`paper-rebalance`。

解决的风险：

- 把投资框架转成可测试、可审计的固定规则。
- 明确第一阶段只使用 sample CSV 和 paper broker。

仍未解决：

- 数据真实性、数据授权、真实成交、真实账户状态、真实资金风险。

### MVP 2: 季度回测引擎

做了什么：

- 增加季度调仓回测、交易成本、滑点、印花税、停牌/涨跌停限制。
- 输出 equity curve、trades、holdings、metrics。

解决的风险：

- 让策略可以在本地样例历史数据上复盘。
- 防止把评分结果直接误认为组合收益。

仍未解决：

- 回测真实性仍然有限，样例数据不能代表真实市场。
- 复权、退市、交易日细节、真实成交价和冲击成本仍未覆盖。

### MVP 3: 数据质量与防未来函数

做了什么：

- 增加 `disclosure_date` 规则、缺失值、stale data、异常值检查。
- 在评分和回测前运行数据质量检查。

解决的风险：

- 避免使用尚未披露的财务数据。
- 数据不合格时拒绝继续评分或回测。

仍未解决：

- 真实 provider 数据修订、跨源不一致、披露时间精度、公告时区和交易日映射。

### MVP 4: 配置化与 Markdown 报告

做了什么：

- 把评分权重、组合约束、交易成本、回测资金、数据质量阈值配置化。
- 增加 run/backtest Markdown 报告。

解决的风险：

- 减少硬编码参数。
- 让运行结果可解释、可复盘。

仍未解决：

- 配置仍只面向本地模拟，不代表可运营配置。

### MVP 5: HTML 报告、通知摘要、审计日志

做了什么：

- 增加 HTML 报告、SVG 图表、dry-run notification summary、decision audit log。

解决的风险：

- 提高复盘可读性。
- 记录关键决策，减少黑箱行为。

仍未解决：

- 通知仅 dry-run，不是生产告警系统。
- 审计留存和审计不可篡改性仍未实现。

### MVP 6: 点时数据增强与因子审计 drilldown

做了什么：

- 增强 point-in-time snapshot。
- 增加 factor audit log 和 `explain-score`。
- 增加本地 data source abstraction 和 schema validation。

解决的风险：

- 每只股票得分可追溯。
- 未来数据源接入前先有点时数据约束。

仍未解决：

- 真实数据源的披露时间、修订历史和字段稳定性仍未验证。

### MVP 7: Adapter Contract 测试层

做了什么：

- 定义 DataProvider contract。
- 增加 `MockAShareProvider`、字段映射、标准化和 provider contract validation。

解决的风险：

- 未来 provider 接入前先有契约测试。
- 避免真实 provider 直接进入主路径。

仍未解决：

- 没有真实 provider SLA、授权、限流、故障模式、字段漂移验证。

### MVP 8: Schema Versioning 与 Contract Diff

做了什么：

- 增加 provider contract YAML、baseline/candidate diff、breaking/additive/compatible 分类。

解决的风险：

- 数据字段变化可以被显式发现和阻断。

仍未解决：

- 合约变化后的真实迁移执行和历史兼容验证仍未覆盖。

### MVP 9: Schema Migration Metadata 与 Diff 可视化

做了什么：

- 增加 migration plan、字段 deprecation 生命周期、兼容窗口、HTML diff visualization。

解决的风险：

- 字段变化不仅能发现，还能解释迁移路径和兼容状态。

仍未解决：

- migration metadata 不参与交易决策，也不替代真实数据校验。

### MVP 10: Pre-Live Safety Gate

做了什么：

- 增加 execution policy、pre-live safety gate、simulate-live-request。
- 默认阻断 live trading、real broker、real order、real network data。

解决的风险：

- 即使未来有人配置错误，也有安全闸门拦截。

仍未解决：

- 安全闸门是本地保护层，不是完整实盘风控系统。

### MVP 11: System Health 与 Release Checklist

做了什么：

- 增加 system health、sensitive scan、artifact manifest、release checklist、release notes draft。
- 固定 v0.11.0 release candidate 的本地复验流程。

解决的风险：

- 发布前可以检查安全默认值、输出产物、文档边界和本地报告。

仍未解决：

- 不自动发布，不自动打 tag，不替代人工 code review 和 release review。

## 系统边界

- 不提供投资建议。
- 不保证收益。
- 不支持真实交易。
- 不连接真实券商。
- 不连接真实网络数据源。
- 不保存真实 broker credentials。
- 不提供荐股功能。
- 不做收益承诺。
- 不做自动实盘下单。
- 不把 mock provider 当成实际行情源。

## 当前最重要结论

1. 这是一个研究型和模拟型系统骨架，不是实盘系统。
2. 当前最有价值的是安全边界、可复验流程、审计和数据接入前的防错结构。
3. 不应因为已有模块较多就误判为可实盘。
4. 安全闸门不是装饰，而是默认阻断真实交易。
5. 下一阶段不应直接接真实 broker，也不应让真实网络数据源直接进入评分和回测主路径。
