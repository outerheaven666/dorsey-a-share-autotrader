# v0.11.0 Pre-Real-Data Gap Analysis

本文列出接真实数据源和真实 broker 前的缺口。结论先行：当前不应该直接接真实 broker，真实数据源也不应直接进入评分和回测主路径。下一步更合理的是 offline real-data fixture import / data vendor export import，而不是在线 API 接入。

## 真实数据接入前缺口

### 数据授权和合法来源

需要先确认数据来源、使用范围、存储范围、再分发限制、研究用途和自动化用途是否合法。没有授权边界，就不应接入。

### Provider SLA

需要评估 provider 的稳定性、延迟、维护窗口、限流、故障恢复和历史数据可追溯性。当前 mock provider 不覆盖这些问题。

### 字段稳定性

真实 provider 可能改字段名、字段类型、单位、枚举值和缺失值表达。现有 schema diff 可以发现变化，但真实字段稳定性仍需 offline fixture 验证。

### 复权 / 未复权处理

回测价格必须明确前复权、后复权、不复权的口径。当前 sample data 没有覆盖除权除息、送转、分红和复权链。

### 停牌 / 涨跌停 / 退市 / 更名

当前只模拟停牌和涨跌停基础状态。真实数据需要处理退市、长期停牌、股票更名、代码变化、板块变化和交易状态枚举。

### 财报披露时间

`disclosure_date` 是核心字段，但真实数据需要更精细地处理披露时间、公告修订、交易日可见性和公告发布后的可交易时间。

### Point-in-time correctness

真实数据必须保证任意 `as_of_date` 下只能看到当时已披露的数据。修订后的财报不能覆盖历史可见快照。

### 数据延迟和修订

真实 provider 可能补发、修订或回填历史数据。系统需要版本化数据快照和修订审计。

### 缺失值策略

需要区分真正缺失、不适用、尚未披露、provider 未覆盖和字段废弃。不同原因应有不同阻断或 warning 策略。

### 异常价格处理

需要处理零价格、极端价格、成交量为零、复权错误、单位错误和涨跌幅异常。当前异常值检查只是样例级。

### 多 provider 对账

如果未来使用多个来源，需要定义主源、备源、对账字段、差异阈值、冲突处理和审计记录。

## 真实 broker 接入前缺口

### 账户权限

必须明确账户权限、可交易市场、可交易品种、资金权限和人工授权范围。

### Read-only mode

任何真实 broker 接入前，应先实现只读账户查询，不允许下单。只读模式应能查询现金、持仓、委托、成交和账户状态。

### Order preview

真实订单前必须有 order preview，展示订单、数量、价格假设、费用、现金影响、持仓影响和风险检查结果。

### Order throttling

需要限频、限额、单笔金额上限、日内订单上限和异常重试控制。

### Kill switch

必须有全局 kill switch，并且默认关闭真实交易路径。

### Manual approval

真实订单必须经过人工审批流程。自动交易不应是下一阶段目标。

### Position reconciliation

需要对 broker 持仓、内部持仓、目标组合、未成交委托进行对账。

### Cash reconciliation

需要对可用现金、冻结资金、资金流水、费用和分红入账进行对账。

### Order status reconciliation

真实 broker 会有已报、已成、部成、废单、撤单、拒单等状态。系统需要可靠状态机和补偿逻辑。

### Reject handling

拒单不能简单跳过，必须分类记录、触发人工检查，并阻止继续扩大风险。

### Audit retention

真实 broker 审计需要保留更长周期，并防止审计记录被无意覆盖。

## 现有模块可以复用的部分

- `adapter contract`: 可作为 future provider 接入前的字段契约入口。
- `schema diff`: 可用于检测 provider 字段变化。
- `schema migration`: 可用于记录字段迁移计划和兼容窗口。
- `pre-live safety gate`: 可继续作为默认阻断层。
- `system health`: 可继续检查安全默认值和本地产物。
- `release checklist`: 可作为每次变更前的人工复验入口。

## 必须重做或增强的部分

- `data loader`: 从 sample loader 升级为可处理 provider export 的离线导入器。
- `trading calendar`: 需要真实交易日、特殊交易日、半日市和市场状态。
- `corporate action handling`: 需要复权、分红、配股、送转等处理。
- `backtest realism`: 需要更真实的价格撮合、可成交性、容量和冲击成本。
- `risk model`: 需要账户级、组合级、订单级和异常状态风控。
- `transaction cost model`: 需要更细的佣金、税费、滑点和成交失败模型。
- `broker adapter`: 未来也必须从只读、模拟、预览、人工审批开始，不能直接进入下单。

## 明确结论

1. 当前不应该直接接真实 broker。
2. 真实数据源也不应直接进入评分和回测主路径。
3. 下一步应该先做 offline real-data fixture import / data vendor export import。
4. 在线 API 接入不是下一阶段优先项。
5. 即使未来接入真实数据，也应先进入隔离的 import/validation/contract test 流程，而不是直接进入策略主路径。
