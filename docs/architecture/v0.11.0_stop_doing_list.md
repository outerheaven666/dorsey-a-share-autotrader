# v0.11.0 Stop Doing List

这份清单用于避免系统因为“看起来能跑”而过早走向错误方向。v0.11.0 仍然是 research-only、paper/backtest simulation 系统，不是实盘系统。

## 短期不要做

- 不要接 QMT / PTrade。
- 不要做实盘下单。
- 不要保存 broker credentials。
- 不要做自动交易。
- 不要做在线行情 API。
- 不要做收益承诺。
- 不要做荐股功能。
- 不要做复杂机器学习模型。
- 不要做日内高频。
- 不要做多账户管理。
- 不要做移动端或复杂 UI。
- 不要为了“看起来高级”加功能。

## 为什么不要做

### 不要接 QMT / PTrade

当前没有真实账户对账、只读模式、订单预览、人工审批、状态机、拒单处理和 kill switch。接入真实 broker 只会制造风险，不会提高当前阶段质量。

### 不要做实盘下单

当前评分、回测和 paper broker 都是模拟路径。模拟路径通过不等于真实资金安全。

### 不要保存 broker credentials

项目没有真实 broker 安全模型，也没有凭据生命周期管理。任何真实账户材料都不应进入仓库。

### 不要做自动交易

系统还没有足够的数据可靠性、账户对账、异常处理和人工审批流程。自动化越早，错误越难控制。

### 不要做在线行情 API

在线 API 会带来授权、延迟、限流、故障、字段漂移和可追溯性问题。下一步应先做 offline vendor export fixture。

### 不要做收益承诺或荐股功能

本系统不提供投资建议，不保证收益，也不应把研究报告包装成荐股输出。

### 不要做复杂机器学习模型

当前更缺的是数据口径、点时正确性、复权处理、交易现实性和风控边界。复杂模型会放大数据问题。

### 不要做日内高频

项目定位是低频基本面量化。日内高频需要完全不同的数据、撮合、延迟和风控体系。

### 不要做多账户管理

多账户会引入权限、分配、对账、隔离和审计复杂度。当前阶段没有必要。

### 不要做移动端或复杂 UI

报告可读性重要，但复杂 UI 不是当前瓶颈。当前瓶颈是数据和安全边界。

### 不要为了“看起来高级”加功能

功能越多，不代表系统越接近可用。v0.11.0 应该优先保持克制、可审计、可复验。

## 应该优先做

- 数据口径复盘。
- 本地真实数据导入格式设计。
- offline vendor export fixture。
- 更真实的交易成本和滑点模型。
- 样本外回测检查。
- 参数稳定性检查。
- 报告可读性。
- 操作手册。
- 回滚方案。
- 人工审批流程设计。

## 下一阶段建议

下一阶段如果继续推进，建议做 `offline real-data import`，但仍然不接在线 provider，不接真实 broker，不进入实盘路径。

推荐顺序：

1. 设计本地 vendor export 标准目录和字段协议。
2. 把真实数据导出文件作为离线 fixture 导入。
3. 跑 schema validation、point-in-time、data quality、contract diff 和 migration checks。
4. 对比 sample CSV 与 offline export 的字段口径。
5. 增强 backtest realism 和交易成本模型。
6. 写操作手册和回滚方案。

## 安全边界提醒

安全闸门不是装饰，而是默认阻断真实交易。任何绕过安全闸门的实现都不应进入主分支。
